# Выход из контейнера и злоупотребление capabilities: обзор векторов атак

## 1. Область

Эта заметка объясняет две связанные, но разные зоны риска в безопасности контейнеров:
- **Выход из контейнера (Container escape)**  -  атакующий пересекает границу изоляции и получает доступ к ресурсам хоста, namespaces хоста или путям выполнения на хосте.
- **Злоупотребление capabilities (Capability abuse)**  -  атакующий остается внутри namespaces контейнера, но злоупотребляет избыточными Linux-привилегиями для доступа к чувствительным данным хоста, устройствам хоста или опасному функционалу ядра.

---

## 2. Почему контейнеры уязвимы

Контейнеры не являются отдельными ядрами или полной границей безопасности.
Это Linux-процессы, изолированные примитивами ядра, такими как:
- namespaces
- cgroups
- Linux capabilities
- seccomp / LSMs
- ограничения файловой системы и runtime
Атака успешна, когда одна или несколько из этих мер контроля неправильно настроены, их удалось обойти, меры контроля избыточно разрешительны или сломаны из-за уязвимости ядра/runtime.

---

## 3. Что считается выходом из контейнера

Выход из контейнера — это любое действие атакующего, которое пересекает границы изоляции и достигает контекста хоста. Частые примеры:
- подключение к namespaces хоста
- доступ к файловым системам хоста через привилегированные mounts
- использование путей управления container runtime для исполнения на уровне хоста
- злоупотребление файлами управления cgroup так, чтобы ядро хоста выполнило код, контролируемый атакующим
- эксплуатация уязвимостей ядра для получения привилегий, эквивалентных хосту

---

## 4. Основные векторы атак для выхода из контейнера

### 4.1 Переход namespaces в контекст хоста

Атакующий может присоединиться к namespaces хоста напрямую или косвенно, обычно через `setns`, `unshare`, `clone` или инструменты вроде `nsenter`.

### Типичные предусловия
- привилегированный контейнер
- доступ к файловым дескрипторам namespaces хоста
- опасные runtime-флаги, такие как `--pid=host`
- слабость ядра/runtime, позволяющая переход namespaces

### Типичное влияние
- видимость процессов хоста
- видимость файловой системы хоста
- видимость сети хоста
- упрощение дальнейшего повышения привилегий и закрепления

---

### 4.2 Злоупотребление cgroup release_agent

Классическая misconfiguration-based атака на выход — злоупотребление механизмом cgroup v1 `release_agent`.

Важно: этот вектор привязан именно к cgroup v1. В cgroup v1 у иерархии может быть файл `release_agent`: если для cgroup включен `notify_on_release`, ядро запускает указанный helper после освобождения cgroup. При ошибочной выдаче `CAP_SYS_ADMIN`, возможности монтировать cgroup filesystem или менять файлы управления атакующий может подставить путь к контролируемому payload и добиться выполнения на хосте.

cgroup v2 — другой API с единой иерархией и более безопасной моделью delegation; прямого аналога v1 `release_agent` для этого паттерна там нет. Это не означает, что cgroup v2 является полноценной security boundary или что ошибки runtime больше невозможны, но конкретный `release_agent` escape нужно проверять как v1-specific риск.

Для Kubernetes в рабочих средах предпочитайте cgroup v2 на поддерживаемой ОС: Kubernetes считает cgroup v2 stable с `v1.25`, а cgroup v1 deprecated с `v1.35`. Практический базовый уровень: Linux kernel `5.8+`, containerd `1.4+` или CRI-O `1.20+`, systemd cgroup driver и дистрибутив, где cgroup v2 включен по умолчанию. На node проверяйте версию командой `stat -fc %T /sys/fs/cgroup/`: `cgroup2fs` означает v2, `tmpfs` обычно указывает на v1.

### Паттерн атаки
Атакующий монтирует cgroup v1, создает дочернюю cgroup, записывает в файлы управления, такие как:
- `notify_on_release`
- `release_agent`
- `cgroup.procs`
После этого ядро хоста выполняет код, контролируемый атакующим, когда cgroup освобождается.

### Типичные предусловия
- node работает на cgroup v1 или гибридной конфигурации, где доступен v1 controller с `release_agent`
- возможность монтировать или изменять файлы cgroup
- обычно избыточные привилегии, такие как `CAP_SYS_ADMIN`
- возможны ошибки, специфичные для ядра

### Типичное влияние
- выполнение команд на хосте
- путь к полной компрометации хоста

### Сигналы для проверки
- `stat -fc %T /sys/fs/cgroup/` на node возвращает не `cgroup2fs`
- внутри workload видны writable cgroup v1 mounts или файлы `release_agent`
- workload имеет `privileged: true`, `CAP_SYS_ADMIN` или host mount к `/sys/fs/cgroup`
- kubelet на Kubernetes `v1.35+` явно запускается с отключенным `failCgroupV1`

---

### 4.3 Злоупотребление Docker socket / runtime API

Если контейнер может обращаться к `docker.sock` или открытому API container runtime, он часто может попросить runtime создать новый привилегированный контейнер или примонтировать файловую систему хоста.

### Паттерн атаки
Атакующий использует runtime API, чтобы запустить дочерний контейнер с:
- повышенными привилегиями
- root хоста, примонтированным в контейнер
- широким доступом к устройствам
- ослабленными мерами безопасности

### Типичные предусловия
- примонтирован `docker.sock`
- экспонирован Docker API
- аналогичное раскрытие канала управления runtime

### Типичное влияние
- быстрое повышение до доступа к файловой системе хоста
- создание последующих привилегированных контейнеров
- закрепление через workload'ы, управляемые runtime

---

### 4.4 Доступ к host `/proc` из привилегированных контейнеров

Привилегированный или избыточно разрешенный контейнер может читать информацию хоста через `/proc`, включая:
- `/proc/1/root/*`
- `/proc/1/environ`
- `/proc/1/maps`
- параметры ядра, такие как `core_pattern`

Раскрытие host `/proc` опасно не само по себе, а тем, что оно превращает изоляцию контейнера в источник рабочей разведки и credential harvesting. `hostPID: true` делает процессы node видимыми внутри контейнера; host mount `/proc` или доступ через `/proc/<pid>/root` может показать filesystem view выбранного host process; `CAP_SYS_PTRACE` и сходные права помогают пройти ptrace-based проверки доступа к proc-файлам вроде `environ` и `maps`.

Что это дает атакующему на практике:
- чтение initial environment процессов, где часто остаются cloud credentials, API tokens, proxy credentials и internal endpoints;
- просмотр host filesystem через `/proc/<pid>/root`, если выбранный процесс находится в нужном mount namespace;
- определение запущенных agents, kubelet/container runtime путей, runtime sockets, security tooling и конфигурационных файлов;
- получение memory layout и путей к binaries/libraries из `maps`, что помогает подбирать дальнейшую эксплуатацию;
- поиск kubelet material, cloud-init данных, node identity и других артефактов, которые можно использовать против Kubernetes API или cloud API.

### Типичные предусловия
- `--privileged`
- `--pid=host`
- `CAP_SYS_PTRACE`
- видимость host `/proc`
- слабая runtime-изоляция

### Типичное влияние
- сбор учетных данных
- обнаружение токенов
- разведка процессов хоста
- подготовка закрепления или lateral movement

### Сигналы для проверки
- Pod использует `hostPID: true` или монтирует host `/proc`, `/sys`, `/run`, `/var/lib/kubelet`, `/var/run` через `hostPath`
- container securityContext содержит `privileged: true`, `CAP_SYS_PTRACE` или `CAP_SYS_ADMIN`
- runtime telemetry фиксирует чтение `/proc/*/environ`, `/proc/*/root`, `/proc/*/maps`, `/proc/sys/kernel/core_pattern`
- audit/runtime события показывают запуск `nsenter`, `chroot`, `find`/`grep` по host paths или обращение к runtime sockets после чтения `/proc`

---

### 4.5 Эксплуатация уязвимых подсистем ядра

Атакующий может эксплуатировать баги ядра, достижимые из контейнера, например через:
- `splice`
- `fsopen` / `fsconfig`
- пути netfilter / netlink
- пути гонок symlink
- попытки перезаписи runtime, такие как `/proc/self/exe`

### Типичные предусловия
- уязвимое ядро или runtime
- достижимая поверхность syscalls
- часто дополнительные условия по привилегиям или namespaces в зависимости от CVE

### Типичное влияние
- повышение привилегий
- выход из namespaces
- компрометация учетных данных
- выполнение на уровне хоста

## 5. Что такое злоупотребление capabilities

Злоупотребление capabilities отличается от полного выхода:
- Атакующий может оставаться внутри namespaces контейнера, но все равно выполнять опасные действия, потому что контейнеру выданы избыточные Linux capabilities или широкие runtime-привилегии.
Во многих средах это более реалистично, чем «чистый» выход, потому что привилегии часто выдаются с избытком ради удобства.

### 5.1 Capability sets: почему `drop`/`add` не вся модель

Linux capabilities не являются одним плоским списком. У процесса есть несколько capability sets, а у исполняемого файла могут быть file capabilities. Ядро проверяет не "что написано в Kubernetes YAML", а фактическое состояние capabilities у конкретного потока выполнения.

Упрощенная модель:

```text
Bounding
└── Permitted
    └── Effective

Ambient ⊆ Permitted ∩ Inheritable
```

Ключевые sets:
- `Effective` - capabilities, которые прямо участвуют в текущих проверках прав ядром.
- `Permitted` - верхняя граница capabilities, которые поток может перевести в `Effective`.
- `Bounding` - потолок capabilities, которые можно получить через `execve`, file capabilities или setuid-root путь.
- `Inheritable` - материал для передачи capabilities через `execve`; сам по себе не дает прав.
- `Ambient` - способ сохранить capabilities после `execve` для непривилегированных файлов; очищается при запуске privileged файла, например setuid или файла с file capabilities.

File capabilities хранятся на исполняемом файле, обычно в xattr `security.capability`. При `execve` ядро пересчитывает capability sets нового процесса на основе sets текущего потока и file capabilities. Поэтому две конфигурации, которые выглядят одинаково на уровне `capabilities.add/drop`, могут вести себя по-разному после запуска entrypoint, `setuid`, `su-exec`, wrapper-скриптов или бинаря с file capabilities.

Практический вывод для Kubernetes:
- `securityContext.capabilities.drop/add` является абстракцией CRI/runtime над несколькими Linux capability sets; фактическое поведение проверяйте в runtime, а не только в YAML.
- `drop: ["ALL"]` должен убирать capability из Effective/Permitted/Bounding; если capability остается в Bounding или возвращается через file capability, принцип наименьших привилегий не достигнут.
- `allowPrivilegeEscalation: false` включает `no_new_privs` и блокирует получение новых привилегий через `execve`, включая setuid и file-capability путь. Это не заменяет `drop: ["ALL"]`, но закрывает важный класс возврата capabilities.
- Kubernetes обычно не выставляет Ambient capabilities. Если приложение сначала стартует с capability, затем делает `setuid`/drop privileges и ожидает сохранить capability после `execve`, оно может сломаться. Это не повод выдавать широкие capabilities; это повод изменить модель запуска или явно документировать исключение.

Проверка внутри Pod:

```bash
kubectl exec -n <ns> <pod> -- sh -c "grep -E 'Cap(Inh|Prm|Eff|Bnd|Amb)|NoNewPrivs' /proc/1/status"
```

Что должно попасть в подтверждение:
- `CapEff`, `CapPrm`, `CapBnd`, `CapAmb` для основного процесса;
- подтверждение `NoNewPrivs: 1`, если policy требует `allowPrivilegeEscalation: false`;
- список file capabilities внутри image для исполняемых файлов, которым они действительно нужны;
- объяснение, почему capability нельзя заменить непривилегированным портом, init-time подготовкой, Kubernetes volume/CSI, sidecar/agent или отдельным изолированным workload.

## 6. Основные векторы атак при злоупотреблении capabilities

### 6.1 Избыточная `CAP_SYS_ADMIN`

### Что это позволяет
В зависимости от контекста может позволять:
- монтирование файловых систем
- управление namespaces
- взаимодействие с чувствительными интерфейсами ядра
- широкие административные действия, которые никогда не должны быть нормой для большинства workload'ов

### Примеры злоупотребления
- монтирование файловых систем, связанных с хостом
- манипуляции со структурами cgroup
- создание путей доступа к чувствительным ресурсам ядра или хоста

### Влияние
Даже без пересечения namespaces это может дать доступ к данным хоста или прямые пути к дальнейшему повышению привилегий.

---

### 6.2 `CAP_SYS_PTRACE` при видимости host PID

### Примеры злоупотребления
- чтение `/proc/1/environ`
- чтение `/proc/1/maps`
- инспекция состояния процессов и раскладки памяти
- подключение к host process или чтение proc-файлов, доступ к которым регулируется ptrace access checks

### Почему это важно
Переменные окружения часто содержат:
- cloud credentials
- API tokens
- service secrets
- внутренние endpoints

`/proc/<pid>/maps` обычно не раскрывает секреты напрямую, но показывает memory mappings, пути к binaries/libraries, heap/stack regions и deleted file mappings. Для атакующего это снижает неопределенность при post-exploitation: проще понять, какие процессы важны, какие версии библиотек используются и где искать следующий слабый контроль.

### Влияние
Это один из самых ценных векторов кражи учетных данных и сильный фактор для lateral movement.

---

### 6.3 `CAP_NET_RAW` и доступ к raw sockets

### Примеры злоупотребления
- захват пакетов
- ARP и разведка интерфейсов
- низкоуровневое сетевое взаимодействие, не ожидаемое для обычных контейнеров приложений

### Влияние
- сетевая разведка
- возможности перехвата учетных данных
- поддержка lateral movement или злоупотреблений, связанных со spoofing

---

### 6.4 Контейнеры `--privileged`

### Типичные свойства
- все capabilities
- широкий доступ к устройствам
- более слабые ограничения seccomp
- более слабая изоляция AppArmor/LSM в зависимости от конфигурации

### Примеры злоупотребления
- монтирование блочных устройств хоста, таких как `/dev/sda1`
- чтение `/dev/kmsg`
- использование `keyctl`
- взаимодействие с путями, связанными с BPF
- попытки прохода путей загрузки модулей

### Влияние
Привилегированному контейнеру часто не нужен классический выход. У него уже может быть достаточно возможностей для операционной компрометации узла.

---

### 6.5 Доступ к устройствам хоста и чувствительным интерфейсам ядра

### Цели злоупотребления
- `/dev/kmsg`
- kernel keyrings
- BPF interfaces
- пути загрузки модулей

### Влияние
- разведка хоста
- подготовка закрепления
- возможности несанкционированного изменения
- более глубокая post-exploitation

---

## 7. Разница между выходом и злоупотреблением capabilities

### Выход из контейнера
Атакующий пересекает границы изоляции.

Примеры:
- присоединение к namespaces хоста
- использование runtime API для создания привилегированных контейнеров с host mount
- эксплуатация багов ядра для получения контекста хоста

### Злоупотребление capabilities
Атакующий остается в назначенном контейнерном контексте, но этот контекст уже слишком избыточный.

Примеры:
- чтение `/proc/1/environ`
- монтирование чувствительных файловых систем
- использование raw sockets
- доступ к `/dev/kmsg`
- злоупотребление keyrings или путями BPF

### Почему это различие важно
Ревью безопасности, которое ищет только «выход», может пропустить более частую операционную реальность: контейнер формально не вышел наружу, но все равно раскрыл секреты хоста или дал возможность компрометации узла.

## 8. User namespaces как mitigation

User namespaces не устраняют все escape-векторы и не превращают контейнер в полноценную security boundary, но меняют последствия компрометации. При `hostUsers: false` root и capabilities внутри контейнера мапятся в непривилегированный диапазон host UID/GID, поэтому UID `0` в контейнере не равен UID `0` на node.

Это особенно важно для workload'ов, которым исторически требовались root или отдельные capabilities. В Kubernetes `v1.36+` User Namespaces находятся в GA для Linux workload'ов, поэтому `hostUsers: false` становится практическим контролем для рабочих сред снижения blast radius, а не экспериментальной настройкой.

Операционная применимость стала выше из-за ID-mapped mounts: kubelet больше не должен массово выполнять recursive `chown` volume-данных только ради смены видимости UID/GID внутри контейнера. Kernel remapping на mount-time делает этот контроль реалистичнее для stateful и volume-heavy workload'ов.

Даже с user namespaces продолжайте запрещать `privileged: true`, минимизировать capabilities, применять seccomp/LSM и контролировать host namespaces, hostPath и runtime sockets. Этот слой снижает последствия, но не заменяет остальные runtime-меры контроля.

## 9. Kubernetes-specific путь эскалации

В Kubernetes container escape или capability abuse часто становится не только node-level инцидентом. Типовая цепочка выглядит так:
- workload получает host access через `privileged`, host namespace, sensitive `hostPath` или runtime socket;
- атакующий читает kubelet/admin конфиги, ServiceAccount material, runtime metadata или host filesystem;
- полученные учетные данные используются против Kubernetes API;
- impact расширяется до Secret reads, создания workload'ов, lateral movement или persistence через RBAC/admission drift.

При ревью фиксируйте не только факт выхода к host, но и следующий reachable control plane step: какие kubeconfig/token/material доступны, какие API-действия они позволяют и какие audit/runtime события должны сработать. Для безопасной проверки пути атаки используйте отдельный playbook: [kubernetes/adversarial-validation/playbook.ru.md](../adversarial-validation/playbook.ru.md).

## 10. Типичные root causes

Большинство этих векторов атак зависят от одного или нескольких следующих сбоев:
- запуск контейнеров с `--privileged`
- монтирование `docker.sock`
- использование `hostPID`, `hostNetwork` или host filesystem mounts без строгого обоснования
- выдача опасных capabilities, таких как `CAP_SYS_ADMIN`, `CAP_SYS_PTRACE`, `CAP_NET_RAW`
- слабая конфигурация seccomp / AppArmor / SELinux
- уязвимые версии ядра или runtime
- слабый контроль создания daemonset/pod в Kubernetes
- предположение, что контейнеры являются жесткой границей безопасности (то есть уже дают достаточную изоляцию)

## 11. Вопросы для ревью безопасности

При ревью workload задайте вопросы:
- Может ли контейнер достичь namespaces хоста?
- Может ли он обращаться к control plane runtime, например `docker.sock`?
- Может ли он читать host paths в `/proc`?
- Есть ли у него опасные capabilities?
- Является ли он привилегированным?
- Может ли он монтировать файловые системы или получать доступ к блочным устройствам?
- Может ли он создавать raw sockets?
- Может ли он обращаться к интерфейсам, взаимодействующим с ядром, таким как keyrings, BPF или пути загрузки модулей?
- Включен ли `hostUsers: false`, если workload работает на Linux и совместим с user namespaces?
- Подвержена ли версия node/kernel/runtime известным путям breakout?
- Приведет ли компрометация этого контейнера к раскрытию учетных данных хоста или позволит привилегированные последующие действия?
- Какой Kubernetes API impact станет возможен после node-level доступа?
